# 에이전트 만들기

이 절에서는 Hermes Desktop 앱에서 **SOUL.md**, **프로필(Profile)**, **AGENTS.md** 세 가지를 직접 다루며 "나만의 에이전트"를 만들어 봅니다. 셋 다 에이전트가 어떻게 행동할지를 바꾸지만, 적용 범위가 서로 다릅니다.

| 무엇 | 한 줄 요약 |
|------|-----------|
| **SOUL.md** | 프로필의 성격·말투·행동 규칙 (그 프로필의 모든 대화에 적용) |
| **프로필** | SOUL.md·스킬·설정을 통째로 묶은 독립된 에이전트 환경 |
| **AGENTS.md** | 특정 프로젝트 폴더에서만 적용되는 작업 지침 |

<Callout type="info" title="이 실습은 Desktop 앱 기준입니다">
SOUL.md와 프로필은 Desktop 앱의 화면에서 바로 편집·생성합니다. 터미널(CLI)에 익숙하다면 `~/.hermes/SOUL.md` 파일을 편집기로 직접 수정해도 결과는 같습니다.
</Callout>

## 실습 1: SOUL.md로 에이전트 성격 정의하기

**SOUL.md**는 프로필에 새겨지는 시스템 프롬프트이자 페르소나입니다. "어떤 성격인지"부터 "어떻게 행동할지"까지, 여기에 적은 대로 에이전트가 움직입니다.

### 편집 화면 열기

1. 좌측 하단 홈 아이콘 옆의 **`...`(manage profile)** 버튼을 클릭합니다.
2. 프로필 목록에서 현재 프로필(예: **default**)을 선택합니다.
3. 오른쪽에 모델(예: `stepfun/step-3.7-flash:free`), 스킬 개수와 함께 **SOUL.MD** 편집 영역이 보입니다.

기본 상태에서는 안내 주석이 담긴 템플릿이 들어 있습니다. 손대기 전에 아래 요청을 보내, 지금 응답의 분위기를 기억해 두세요.

<TerminalBlock>
```
현재 폴더의 위치를 알려주고, 어떤 파일들이 있는지 살펴봐줘
```
</TerminalBlock>

### 커스텀 SOUL.md 작성

편집 영역의 내용을 아래처럼 바꿉니다. "나는 친절한 AI야" 같은 성격 묘사보다, 지킬 수 있는 **행동 규칙**으로 적는 것이 핵심입니다.

<TerminalBlock title="SOUL.md">
```markdown
# 작업 파트너

나는 한국어로 일하는 실용적인 작업 파트너다.

## 행동 규칙
- 항상 한국어로 답한다
- 동료에게 말하듯 대등하게 말한다
- 모르면 모른다고 분명히 밝힌다
- "물론이죠!", "훌륭한 질문이에요!" 같은 빈말은 쓰지 않는다

## 일하는 방식
- 작업을 시작하기 전에 할 일을 한 줄로 요약한다
- 되돌리기 어려운 명령은 실행 전에 먼저 확인한다
- 오류가 나면 해결책보다 원인 분석을 먼저 내놓는다
- 방법이 여러 개면 장단점을 짧게 비교한다

## 응답 형식
- 결론을 먼저, 설명은 뒤에 붙인다
- 코드에는 짧은 주석을 단다
- 표로 정리할 수 있으면 줄글 대신 표를 쓴다
```
</TerminalBlock>

작성했으면 **Save SOUL.md** 버튼을 눌러 저장합니다.

<Callout type="info" title="새 세션을 열 필요가 없습니다">
SOUL.md는 메시지를 보낼 때마다 새로 읽힙니다. 저장한 뒤 곧바로 다음 메시지를 보내면 바로 반영됩니다. 재시작이나 새 세션은 필요 없습니다.
</Callout>

### 변화 확인

이어서 규칙이 실제로 지켜지는지 아래 질문으로 확인해 봅니다. 한국어로 답하는지, 할 일을 먼저 한 줄로 요약하는지, 표로 정리하는지가 보이면 SOUL.md가 적용된 것입니다.

<TerminalBlock>
```
초전도체가 상온에서 가능한지 지금 기술로 설명해줘
```
</TerminalBlock>

"모르면 모른다고 밝힌다"와 "빈말 금지" 규칙이 지켜지는지 보세요. 과장 없이 확실한 부분과 불확실한 부분을 구분하는지, 장황한 인사 없이 본론으로 들어가는지가 관찰 포인트입니다.

<Callout type="warning" title="관찰 포인트 — SOUL.md">
- SOUL.md는 시스템 프롬프트의 **페르소나 슬롯**입니다. 비워 두면 기본 페르소나(`You are Hermes Agent…`)로 돌아가고, 채우면 그 자리를 여러분이 적은 성격이 대신합니다. 도구 사용·환경 정보 같은 기본 동작은 SOUL.md와 별개로 항상 유지됩니다.
- 핵심 규칙은 간결하게 적을수록 모델이 더 잘 따릅니다. 파일이 지나치게 길면 모델 컨텍스트에 따라 자동으로 일부가 잘릴 수 있습니다.
</Callout>

## 실습 2: 새 프로필로 에이전트 분리하기

**프로필**은 SOUL.md·스킬·설정을 통째로 묶은 독립된 Hermes 환경입니다. 용도가 다른 에이전트를 섞지 않고 따로 둘 수 있습니다. 예를 들어 실습 1의 default는 "작업 파트너"로 두고, 글쓰기 전용 에이전트를 새 프로필로 만드는 식입니다.

### 새 프로필 만들기

좌측 하단에서 두 가지 방법 중 하나로 만듭니다.

- 홈 아이콘 옆 **`+`** 버튼 클릭
- 또는 **`...`(manage profile)** → 좌측 상단 **`+ New profile`** 클릭

**New profile** 창이 열리면 다음을 채웁니다.

1. **Name** — 프로필 이름. 소문자·숫자·하이픈·밑줄만 쓰며 글자나 숫자로 시작해야 합니다. (예: `writer`)
2. **Clone from** — 복제할 원본 프로필. `default`를 고르면 설정·스킬·SOUL.md를 그대로 복사해 옵니다.
3. **SOUL.md (optional)** — 이 프로필만의 페르소나. 비워 두면 복제한 내용을 그대로 쓰고, 지금 새로 적어 넣을 수도 있습니다.

글쓰기 전용 프로필이라면 SOUL.md에 이런 페르소나를 넣어 봅니다.

<TerminalBlock title="SOUL.md">
```markdown
# 글쓰기 도우미

나는 한국어 글을 다듬는 편집자다.

## 행동 규칙
- 항상 한국어로 답한다
- 문장을 고칠 때는 고친 이유를 한 줄로 덧붙인다
- 군더더기 표현과 중복을 먼저 걷어낸다
- 원문의 의도와 어조는 임의로 바꾸지 않는다

## 응답 형식
- 고친 문장을 먼저 보여주고, 변경 이유는 목록으로 정리한다
```
</TerminalBlock>

**Create profile**을 누르면 새 프로필이 만들어집니다.

### 프로필 전환하며 비교

좌측 프로필 목록에서 새로 만든 프로필과 default를 오가며, 같은 요청에 응답이 어떻게 달라지는지 봅니다.

<TerminalBlock>
```
아래 문장을 다듬어줘: 저는 이번에 새롭게 시작하게 된 프로젝트에 대해서 한번 설명을 드려보고자 합니다
```
</TerminalBlock>

<Callout type="warning" title="관찰 포인트 — 프로필">
- 프로필은 **독립된 환경**입니다. 서로 SOUL.md·스킬·설정을 공유하지 않습니다.
- `default`를 Clone 하면 기존 환경을 그대로 복제한 뒤 SOUL.md만 바꿔 실험할 수 있습니다.
- 한 프로필의 SOUL.md를 바꿔도 다른 프로필에는 영향이 없습니다.
- 용도(코딩·글쓰기·리서치 등)별로 프로필을 나눠 두면 성격이 섞이지 않습니다.
</Callout>

## 실습 3: AGENTS.md로 프로젝트별 지침 설정

**AGENTS.md**는 프로젝트 폴더에 두는 작업 지침입니다. SOUL.md가 "이 에이전트가 누구인가(모든 작업에 적용)"라면, AGENTS.md는 "이 폴더에서는 이렇게 일한다(그 프로젝트에만 적용)"입니다.

작업할 폴더에 `AGENTS.md`를 아래 내용으로 만듭니다. 직접 편집기로 작성해도 되고, 에이전트에게 "이 폴더에 AGENTS.md를 이렇게 만들어줘"라고 부탁해도 됩니다.

<TerminalBlock title="AGENTS.md">
```markdown
# 샘플 프로젝트

## 코딩 규칙
- Python 3.11 기준으로 작성한다
- 변수·함수명은 snake_case를 쓴다
- 모든 함수에 한 줄 docstring을 단다
- 출력은 print 대신 logging 모듈을 쓴다
- 함수 시그니처에 타입 힌트를 붙인다
```
</TerminalBlock>

<Callout type="info" title="작업 디렉터리 바꾸는 방법">
우측 상단의 설정 아이콘 클릭 → 왼쪽 **Archived Chats** 메뉴 선택 → **Default project directory** 항목의 **Change** 버튼을 클릭해 원하는 작업 디렉터리로 변경합니다.
</Callout>

그 폴더를 작업 디렉터리로 둔 상태에서 코드를 하나 요청하고, 규칙을 따르는지 봅니다.

<TerminalBlock>
```
CSV 파일을 읽어서 열별 평균을 출력하는 스크립트를 만들어줘
```
</TerminalBlock>

<Callout type="warning" title="관찰 포인트 — AGENTS.md">
- `print` 대신 `logging`을 쓰는지
- 이름을 snake_case로 짓는지
- 함수마다 docstring이 있는지
- 타입 힌트가 붙는지
- AGENTS.md가 없는 폴더에서 같은 요청을 하면 결과가 어떻게 달라지는지도 비교해 보세요.
</Callout>

## 세 가지 역할 정리

| | SOUL.md | 프로필 | AGENTS.md |
|---|---------|--------|-----------|
| **무엇** | 프로필의 페르소나·행동 규칙 | SOUL.md·스킬·설정을 묶은 환경 | 프로젝트 폴더의 작업 지침 |
| **적용 범위** | 그 프로필의 모든 대화 | 선택한 프로필 전체 | 해당 프로젝트 폴더만 |
| **만드는 곳** | 프로필의 SOUL.MD 편집 영역 | `+` 또는 `...` → New profile | 프로젝트 폴더의 `AGENTS.md` 파일 |
| **비유** | 사람의 가치관·말투 | 역할이 다른 또 다른 사람 | 팀의 코딩 컨벤션 |
